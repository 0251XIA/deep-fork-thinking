## 案例一：职业转型困惑 🟢普通

**场景**：工作 5 年，考虑是否转行，但担心风险。

```
【Step 1~3 收集确认】
┌────────────────────────────────┐
│ 目标：找到值得长期投入的方向    │
│ 现状：想转行但担心风险          │
│ 约束：房贷压力                  │
│ 已尝试：暂无                    │
└────────────────────────────────┘
因素：经济压力、目标行业了解、技能迁移性
分叉点：经济压力、目标行业了解

【复杂度评估】
分叉点：2个 ✓
相关方：1个（自己）✓
周期：长期 ✓
利益冲突：无 ✓
可逆性：可逆 ✓
→ 🟢 普通级

【Step 3a~4 分析输出】
最终结果：找到值得投入的方向
│
├─ 经济压力
│  ├─ >12月 → 深度调研+小步试错
│  ├─ 6~12月 → 加速收集+预设退出点
│  └─ <6月 → 先兼职探索，稳住现金
│
└─ 目标行业了解（嵌套）
   ├─ 已调研 → 制定转型计划
   └─ 有限 → 先做1个月信息收集

【迭代验证】✓ 已确认
```

**总结**：先算清能撑多久，再做信息收集。经济压力大绝不裸辞。

## 案例二：客户投诉升级 🔵中复杂

**场景**：客户要求退款+赔偿，责任不全在我方。

```
【Step 1~3 收集确认】
┌────────────────────────────────┐
│ 目标：客户满意但不损公司利益    │
│ 现状：产品质量问题，客户要求高  │
│ 约束：必须在授权范围内处理      │
│ 已尝试：暂无                    │
└────────────────────────────────┘
因素：责任归属、客户影响力、政策灵活度
分叉点：责任归属、客户影响力、政策灵活度

【复杂度评估】
分叉点：3个 ✓
相关方：2个（客户、公司）✓
周期：1~2周 ✓
利益冲突：潜在（赔钱vs满意度）✓
可逆性：部分可逆 ✓
→ 🔵 中复杂级

【Step 3a~4 分析输出】
最终结果：客户满意 + 公司不亏
│
├─ 责任归属
│  ├─ 公司主责 → 全额退款+补偿+内部复盘
│  ├─ 各半 → 部分补偿+设上限+记录
│  └─ 客户主责 → 解释政策+最小善意
│
└─ 客户影响力（嵌套）
   ├─ 高 → 升级处理+领导参与
   └─ 低 → 标准流程

【迭代验证】✓ 已确认
```

**总结**：先定性责任，再看客户影响力，两个维度决定补偿力度。

## 案例三：跨部门协作 🔴高复杂

**场景**：你牵头的项目需要多部门配合，但各部门优先级不同。

```
【Step 1~3 收集确认】
┌────────────────────────────────┐
│ 目标：项目按时上线，各方满意    │
│ 现状：各部门有自己的KPI，不配合  │
│ 约束：Deadline 可能有弹性       │
│ 已尝试：发邮件催促，无效         │
└────────────────────────────────┘
因素：高层支持力度、各部门原优先级、Deadline是否可调、资源是否充足、各部门利益诉求差异、利益绑定机制是否成熟
分叉点：高层支持力度、各部门原优先级、Deadline是否可调、利益绑定机制是否成熟

【复杂度评估】
分叉点：4个 ✓（高层支持、各部门优先级、Deadline、利益绑定）
相关方：≥6人（市场部、技术部、财务部等）✓
周期：≥1月 ✓
利益冲突：明显（部门利益冲突）✓
可逆性：部分可逆
→ 🔴 高复杂级

【Step 3a~4 分析输出】
最终结果：项目上线 + 各方满意
│
├─ 高层支持
│  ├─ 强 → 正式文件+指标分解+定期汇报
│  └─ 弱 → 找利益结合点+让部门自主要参与  ⭐ECPBMR
│
├─ 各部门优先级（嵌套）
│  ├─ 愿意 → 保持沟通+及时认可
│  └─ 不愿 → 展示对他们KPI的帮助
│
├─ Deadline
│  ├─ 可调 → 分阶段+甜点先行
│  └─ 硬 → 集中资源攻关键
│
└─ 利益绑定机制（嵌套）
   ├─ 成熟 → 直接对接KPI
   └─ 不成熟 → 先建立利益绑定，再推动配合

【ECPBMR 框架】（针对"高层支持弱"分支）
E: 项目在Q2前上线
P: 高层支持→部门配合→资源到位
B: 高层背书是关键突破点
M: 本周完成高层汇报，下周启动部门对齐
R: 7天检查点/30天里程碑/60天复盘

【迭代验证】✓ 已确认
```

**总结**：不是"求配合"，而是"让配合变成各部门的最优解"——通过利益绑定机制让跨部门协作从阻力变成动力。

## 案例四：领导安排非本职工作 🔵中复杂

**场景**：领导安排非本职任务，不想做但不想得罪人。

```
【Step 1~3 收集确认】
┌────────────────────────────────┐
│ 目标：拒绝工作 + 不伤和气       │
│ 现状：领导安排非本职任务        │
│ 约束：不清楚 deadline          │
│ 已尝试：暂无                    │
└────────────────────────────────┘
因素：拒绝的代价、领导性格、是否有人更合适、本职是否饱满
分叉点：拒绝的代价、领导性格

【复杂度评估】
分叉点：2个
相关方：2个（领导、自己）✓
周期：不确定（取决于任务）✓
利益冲突：潜在（个人vs组织）✓
可逆性：部分可逆 ✓
→ 🔵 中复杂级

【Step 3a~4 分析输出】
最终结果：拒绝 + 不翻脸
│
├─ 拒绝代价
│  ├─ 高 → 部分接受+设条件
│  └─ 低 → 坦诚说明+替代方案
│
├─ 领导性格（嵌套）
│  ├─ 强势 → 给台阶+转决策权
│  └─ 随和 → 直接沟通+协商
│
└─ 是否有人更合适
   └─ 有 → 推荐，转移矛盾

【迭代验证】✓ 已确认
```

**总结**：不说"不"，说"可以，但在 XXX 前提下"——把决策权交还领导。

## 案例五：产品方向决策 🔴高复杂

**场景**：公司产品面临重大方向调整，团队对未来方向有分歧。

```
【Step 1~3 收集确认】
┌────────────────────────────────┐
│ 目标：确定产品方向，团队共识    │
│ 现状：技术团队主张A方向，业务方主张B方向 │
│ 约束：资源有限，半年内必须出成果 │
│ 已尝试：多次开会讨论无果        │
└────────────────────────────────┘
因素：技术可行性、市场需求、团队能力、战略价值、资源成本、政治因素
分叉点：市场验证度、技术风险、团队共识、高层支持

【复杂度评估】
分叉点：4个 ✓
相关方：≥6人（研发/产品/市场/销售/高管）✓
周期：≥1月 ✓
利益冲突：明显（技术vs业务方向分歧）✓
可逆性：部分可逆
→ 🔴 高复杂级

【Step 3a~4 分析输出】
最终结果：产品方向确定 + 团队共识
│
├─ 市场验证度
│  ├─ 已验证 → 选择方向A，资源集中投入
│  └─ 未验证 → 先做MVP验证，再决定
│
├─ 技术风险
│  ├─ 低 → 可快速推进
│  └─ 高 → 需要更多时间和资源  ⭐ECPBMR
│
├─ 团队共识
│  ├─ 高 → 统一执行
│  └─ 低 → 先做对齐工作，再推进
│
└─ 高层支持
   ├─ 强 → 获得更多资源保障
   └─ 弱 → 寻找小切口突破

【ECPBMR 框架】（针对"技术风险高"分支）
E: 半年内完成产品方向验证
P: 市场验证→技术选型→MVP开发→用户反馈
B: 技术风险是核心变量，必须先解决
M: 本月完成技术可行性评估，下月启动MVP
R: 2周技术验证/4周MVP/8周用户测试

【迭代验证】✓ 已确认
```

**总结**：当技术风险是主要矛盾时，先用最小成本验证技术可行性，再决定是否大规模投入。
