# 深度分叉法（Deep Fork Thinking）

## 是什么

深度分叉法是一款结构化决策框架，专为复杂问题分析而设计。它融合了分叉法的问题拆分逻辑与三大经典思维工具，让使用者能够系统性地梳理问题、挖到本质、并输出可执行的行动方案。

核心口诀：**以终为始找分叉，追问三层挖本质，剃刀出鞘做减法，树状输出可落地。**

---

## 能解决什么问题

| 场景 | 示例 |
|------|------|
| 问题杂乱无头绪，不知从何下手 | 新项目涉及多方利益，不知道先处理哪个 |
| 多因素相互缠绕，理不清关系 | 职业选择受薪资、发展、兴趣多方拉扯 |
| 需要帮他人理清思路 | 下属遇到困境，需要引导他自行找到方向 |
| 决策后果重大，需要系统分析 | 转型、跳槽、合作等不可逆决定 |

---

## 使用了哪些专业方式

**1. 苏格拉底追问**
通过逐层追问（为什么→前提成立吗→更深原因→根本矛盾），挖到问题的本质原因而非表象。每个分叉点追问 1~5 层，直到触及本质或公理。

**2. 第一性原理**
抛开现有流程和惯例，从零开始思考"最基础的逻辑是什么"。用于剥离思维惯性，识别真正必要的行动。

**3. 奥卡姆剃刀**
用「能一步完成的不用两步、能直接验证的不绕弯子、能合并的分支合并」原则精简决策树，确保输出简洁可落地。

**4. 分叉法**
将复杂问题拆解为多个「取值不同则做法不同」的关键岔路口，以树状结构组织决策路径。

**5. ECPBMR 执行框架（仅高复杂场景）**
针对「路径断裂点」——即该分支取值决定了后续所有分支是否有意义——提供完整的执行方案：终局(End)、路径(Path)、突破(Breakthrough)、执行(Move)、复盘(Review)。

## 流程是怎样的

### 整体架构

```
澄清阶段（确认制）                    分析阶段（推理制）
┌─────────────────────┐               ┌─────────────────────┐
│ Step 1  收集四要素   │               │ Step 3a 追问分叉点  │
│       ↓             │               │       ↓             │
│ Step 2  确认目标因素 │               │ Step 3b 挖本质      │
│       ↓             │               │       ↓             │
│ Step 3  识别分叉点   │               │ Step 4  奥卡姆剃刀   │
└─────────────────────┘               │       ↓             │
                                        │ 输出决策树 + 总结   │
                                        └─────────────────────┘
```

### Step 1：收集四要素

通过四个问题收集信息：
- **目标**：最终想要什么结果？
- **现状**：当前什么情况，最困扰的一点是什么？
- **约束**：硬性限制（时间、预算、权限、风险底线）？
- **已尝试**：做过什么尝试，结果如何？

### Step 2：确认目标与因素

确认目标是否准确，列出影响因素列表。

### Step 3：识别分叉点

对每个因素提问：「如果这个因素为 A 或为 B，接下来的做法会不同吗？」

只有取值不同导致做法不同时，才构成真正的分叉点。

### 复杂度评估

根据分叉点数量、相关方人数、周期、利益冲突、可逆性五维度，评估问题属于哪个等级：

| 等级 | 分叉点 | 相关方 | 周期 | 输出 |
|------|--------|--------|------|------|
| 🟢 普通 | 1~2 个 | ≤2 人 | ≤7 天 | 简化 |
| 🔵 中复杂 | 2~3 个 | 3~5 人 | 1~4 周 | 标准 |
| 🔴 高复杂 | ≥4 个 | ≥6 人 | ≥1 月 | 完整 + ECPBMR |

### Step 3a：追问（按复杂度调整深度）

对每个分叉点进行苏格拉底追问：
- 第1轮：为什么这个因素会影响结果？
- 第2轮：这个前提成立吗？如果不成立，本质是什么？
- 第3轮：还有什么更深的原因？换一个人做会不同吗？
- 第4轮+：这反映了什么根本矛盾？

**终止标准（客观判断）**：
- 🟢 普通：用户回答出现"不一定""未必"
- 🔵 中复杂：用户回答出现"本质是""核心是"
- 🔴 高复杂：用户开始用"规则/流程/制度"来解释，或明确指出是组织层面的限制

### Step 3b：挖本质（针对核心分叉点）

- 抛开现有流程，最基础的原理是什么？
- 如果从零开始设计，我会怎么做？
- 现有做法有多少是"一直这么做"而非"必须这么做"？

### Step 4：奥卡姆剃刀 + 决策树输出

精简合并分支，每条路径输出「条件 → 本质 → 3 步以内行动」。

**输出格式**：
```
最终结果：[目标]
│
├─ 分叉点1：[因素名]
│  ├─ 取值A → 本质：[本质] → 行动：[1. 2. 3.]
│  └─ 取值B → 本质：[本质] → 行动：[1. 2. 3.]
│
└─ 分叉点2：[因素名]（嵌套）
   └─ ...
```

### 总结概括（必须执行）

输出决策树后，必须进行总结：
- **背景**：问题的来龙去脉（1~2句话）
- **核心发现**：通过追问挖到的本质，不是表象
- **关键洞察**：2~3个最重要的判断
- **行动方向**：用户应该怎么做

## 有什么价值

| 价值点 | 说明 |
|--------|------|
| 结构化 | 避免遗漏关键因素，确保分析完整 |
| 深入本质 | 通过追问层层剥开表象，找到真正驱动问题的原因 |
| 可执行 | 每条分支都有明确的行动步骤，≤3步，可直接落地 |
| 适配不同复杂度 | 从简单到复杂，都有对应的处理深度 |
| 验证机制 | 输出后有迭代确认，确保分析符合实际情况 |

## 一句话总结

用分叉法画地图找岔路口，用三大工具在每个岔路口挖到地心，最后只带最轻的行囊上路。
