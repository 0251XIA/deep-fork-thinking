# 深度分叉法（Deep Fork Thinking）

## 核心逻辑

```
澄清阶段（确认制）                    分析阶段（推理制）
┌─────────────────────┐               ┌─────────────────────┐
│ Step 1  收集四要素   │               │ Step 3a 追问分叉点  │
│       ↓             │               │       ↓             │
│ Step 2  确认目标因素 │               │ Step 3b 挖本质      │
│       ↓             │               │       ↓             │
│ Step 3  识别分叉点   │               │ Step 4  奥卡姆剃刀   │
└─────────────────────┘               │       ↓             │
                                        │ 输出决策树          │
                                        └─────────────────────┘
```

**口诀**：以终为始找分叉，追问三层挖本质，剃刀出鞘做减法，树状输出可落地。

**三大工具调用**：Step 3a 苏格拉底追问 → Step 3b 第一性原理 → Step 4 奥卡姆剃刀

---

## 何时使用

| ✅ 适用 | ❌ 不适用 |
|----------|----------|
| 问题杂乱无头绪 | 简单二选一 |

| 需输出可落地方案 |  |

## 澄清阶段（Step 1~3）

> ⚠️ **澄清阶段采用确认制**：每一步AI输出结果后，需要用户确认才能进入下一步。如果用户推翻之前的结论，退回对应步骤重新确认，不需要从头开始。

### Step 1：收集四要素 + Fallback 预备 📋

**提问**：请回答以下问题：

> **1. 目标**：您最终想要什么结果？
> **2. 现状**：当前是什么情况？最困扰您的一点是什么？
> **3. 约束**：有什么硬性限制（时间、预算、权限、风险底线）？
> **4. 已尝试**：您或他人已经做过什么尝试？结果如何？

**Fallback（适用于整个澄清阶段）**：

| 用户表现 | 我的回应 |
|----------|----------|
| "都行/随便" | "那您不想要哪种结果？排除法试试？" |
| 说不清楚 | "能举个具体例子吗？" |
| 一直绕圈子 | "我们从头理一下——最初想解决什么？" |
| 情绪激烈 | "我理解您着急。为了解决，您希望我先做什么？" |
| 沉默 | "我说清楚了吗？有没有需要补充的？" |
| 多个问题同时出现 | "我看到您提到了几个问题。我们先从最重要的一个开始，您觉得哪个最紧急？" |
| 推翻之前结论 | "理解了。您想调整的是哪一部分？我重新确认。" |

**回退机制**：用户推翻之前结果时，退回对应步骤重新确认，不需要从头开始。

**输出摘要**：

```
┌─────────────────────────────────────────┐
│ 目标：_______________________________   │
│ 现状：_______________________________   │
│ 约束：_______________________________   │
│ 已尝试：_____________________________   │
└─────────────────────────────────────────┘
```

**确认后** → 进入 Step 2

---

### Step 2：确认目标与因素 🎯🔍

**输出**：

```
目标：________

影响因素：
1. ________
2. ________
3. ________
```

**确认话术**：`目标是否准确？影响因素是否涵盖关键点？需要补充或修改吗？`

**确认后** → 进入 Step 3

---

### Step 3：识别分叉点 ⚡

**判断标准**：如果这个因素为 A 或为 B，接下来的**做法会不同**吗？

**输出**：

```
分叉点排序：
1. [因素A] — 例：A情况下做X，B情况下做Y
2. [因素B] — 例：B情况下做X，A情况下做Y
```

**确认话术**：`这X个分叉点是否准确反映了关键岔路口？优先级排序是否符合您的判断？`

---

## 复杂度分级

在 Step 3（澄清阶段）完成后评估：

| 维度 | 🟢 普通 | 🔵 中复杂 | 🔴 高复杂 |
|------|---------|-----------|-----------|
| 分叉点数量 | 1~2 个 | 2~3 个 | ≥4 个 |
| 相关方 | ≤2 人 | 3~5 人 | ≥6 人 |
| 周期 | ≤7 天 | 1~4 周 | ≥1 月 |
| 利益冲突 | 无 | 潜在 | 明显 |
| 可逆性 | 可逆 | 部分可逆 | 不可逆 |

**判定**：满足 ≥3 项则按对应等级处理。

**加载**：
- 🟢 普通：`./references/普通级.md`（追问 1~2 层）
- 🔵 中复杂：`./references/中复杂级.md`（追问 2~3 层）
- 🔴 高复杂：`./references/高复杂级.md`（追问 3~5 层）

## 分析阶段（Step 3a~4）

### Step 3a：苏格拉底追问（对每个分叉点）

**追问框架**（逐轮追问 + 随时检查终止）：
- 第1轮：为什么 [分叉点] 会影响结果？
- 第2轮：这个前提成立吗？如果不成立，本质是什么？
- 第3轮：还有什么更深的原因？
- 第4轮+：这反映了什么根本矛盾？
（追问轮数根据等级调整：普通1~2轮，中复杂2~3轮，高复杂3~5轮；每轮结束后检查终止条件是否满足）

**追问终止判断（客观标准）**：

| 等级 | 终止条件 | 客观判断标准 |
|------|----------|-------------|
| 🟢 普通 | 找到反例 | 用户回答中出现"不一定""如果...就不同了""未必" |
| 🔵 中复杂 | 触及本质 | 用户回答中出现"本质是...""归根结底是...""核心是..." |
| 🔴 高复杂 | 触及事实/公理 | 用户无法再追问更底层的原因（开始用"规则/流程/制度"来解释，而非继续回答"为什么"）；或明确指出这是组织/制度层面的限制（"这是规定""这是政策/法律限制""这是权力结构决定"） |

**追问终止执行**：
1. 每轮追问后检查用户回答是否满足终止条件
2. 如果满足，停止追问，进入下一个分叉点
3. 如果不满足，继续追问（不超过该等级最大层数）

---

### Step 3b：第一性原理

（在所有分叉点追问完成后，针对核心分叉点执行）

**追问**：
- 抛开现有流程/惯例，最基础的原理是什么？
- 如果从零开始设计，我会怎么做？
- 现有做法有多少是"一直这么做"而非"必须这么做"？

**示例**（统一分叉点：任务价值判断）：
```
Q: 抛开现有的价值判断方式，这件事最基本的逻辑是什么？
A: 投入产出比——值不值得做

Q: 如果从零开始设计，你会怎么做？
A: 先问"做完对我有什么实际好处"，再问"不做会失去什么"

Q: 现有做法有多少是"一直这么做"而非"必须这么做"？
A: 大部分"应该这样做"都是惯例而非最优解
```

---

### Step 4：奥卡姆剃刀 + 决策树 🌲

**剃刀原则**：
- 能一步完成的不用两步
- 能直接验证的不绕弯子
- 能合并的分支合并

**决策树输出**：

```
最终结果：[目标]
│
├─ 分叉点1：[因素名]
│  ├─ 取值A → 本质：[本质] → 行动：[1. 2. 3.]
│  └─ 取值B → 本质：[本质] → 行动：[1. 2. 3.]
│
└─ 分叉点2：[因素名]（嵌套）
   └─ ...
```

**每分支**：`条件 → 本质 → 3 步以内行动`

**🔴 高复杂附加**：在核心分叉点（影响路径关键节点的分支）标注 `⭐ECPBMR`，在决策树下方输出对应的 ECPBMR 框架。

**⭐ECPBMR 嵌入规则**：
- 适用于影响路径中关键节点的分支（如"高层支持弱"是突破点）
- 判断标准：该分支的取值决定了后续其他分支是否还有意义
- 不是所有分支都标注，只标注"路径断裂点"

---

## Step 4 后：总结概括

> ⚠️ **无论复杂度等级，输出决策树后必须进行总结概括**

**总结话术**：
> 让我来总结一下这次分析的核心内容：
>
> **背景**：……（问题的来龙去脉）
>
> **核心发现**：……（通过追问挖到的本质，不是表象）
>
> **关键洞察**：……（2~3个最重要的判断）
>
> **行动方向**：……（用户应该怎么做）

**总结格式**：
```
【总结】
背景：……
核心发现：……
关键洞察：……
行动方向：……
```

**原则**：
- 总结的是分析过程得出的**结论**，不是重复决策树
- 背景要简洁，1~2句话交代清楚
- 核心发现要回答"本质是什么"
- 关键洞察要有观点，不是复述
- 行动方向要具体可执行

---

## 迭代验证机制

**决策树输出后，必须执行以下确认**：

**确认话术**：
> "以上是我的分析。这个决策树是否符合您的情况？有没有遗漏的分支或需要调整的地方？"

**追问确认示例**（用于澄清阶段每步结束后）：
> Step 1 完成后："我整理一下——您的目标是XX，现状是XX，约束是XX，已尝试的是XX。对吗？"
> Step 2 完成后："目标是XX，影响因素有：1/2/3。您看有没有遗漏或需要调整的？"
> Step 3 完成后："我识别出X个分叉点：1/2/3。这个排序是否符合您的判断？"

**用户反馈处理**：
- 如果用户确认 → 完成
- 如果用户修改 → 退回对应的分叉点重新分析，不需要重新走 Step 1~3
- 如果用户推翻目标 → 退回到 Step 2 重新确认目标
- 如果用户补充新信息 → 评估是否影响分叉点，如影响则重新评估复杂度
- 如果用户质疑分析框架 → "我理解您对XX有疑虑。请告诉我您的顾虑是什么，我重新梳理。"

## 输出质量检查

- [ ] 每个分叉点的取值是否互斥且完备？
- [ ] 每个分支的本质是否不同？
- [ ] 每个分支的行动是否 ≤3 步？
- [ ] 是否存在"空分支"？

## 一句话总结

从目标出发，找到关键岔路口，用追问挖本质，用剃刀砍冗余——最后得到一张可落地的决策地图。
